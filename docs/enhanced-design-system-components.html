Component-Level Content System

Consistency, Clarity, and Accessibility Across Truist Commercial Interfaces

Component Syntax + Design Library

This table includes content syntax, voice/tone, accessibility, localization notes, and rationale for each UI component.

Component Content Syntax Example Voice & Tone Char Count Guidance A11y Notes Localization Notes Usage Guidelines Context Type Rationale
Status Banner [X] [items] need your [action] 1 payment requires your approval Direct, action-oriented. Use urgency without panic. Reassuring when confirming success. Max 10 words / 60 characters Use ARIA live regions. Ensure contrast and icon labels. Check plural rules. Adapt urgency language per locale. Top-of-page priority. One banner per screen. Use for time-sensitive or success states. Standard / High-Stakes
CTA Button [Verb] [Object] Submit for Approval Confident, outcome-driven. Prioritize clarity over cleverness. 2–4 words / Max 25 characters Match visual label to accessible label. Use aria-label if needed. Ensure verb translation remains short and actionable. Primary: irreversible actions. Secondary: navigation, cancel. One primary CTA per screen. Standard
Quick Action Tile [Action Verb]\n[Action Type / Help Text]\n[Optional Status Tag] Schedule\nRecurring payments\n⚡ Demo Ready Verb-first, confident. Help text should be informative but light. Label: 1–2 words; Subtext: max 25 characters Entire tile must be keyboard accessible. Use `aria-label` for embedded statuses. Make sure status badges translate cleanly. Use icon + label combo. Indicate demo/limited access with status badge or note. Task Entry / CTA
Summary Card (Recent Payment) [Vendor Name]\n[Category • Frequency/Due Date]\n[Optional Status • CTA] Downtown Property\nOffice Rent • Due Jan 31\n⚡ Pending manager approval Professional and scannable. Plain English. Avoid acronyms. Vendor: 35 chars max; Details: 1–2 lines Icons must have alt text. Ensure high contrast for details/status. Allow space for long translated vendor or status lines. Stack content vertically. Truncate long vendor names. Show urgency/status when relevant. Standard / Informational
Navigation Label [Section Name] Payments, History, Dashboard Short, noun-based. No verbs unless required. Max 12 characters preferred Label active state. Use semantic nav roles. Account for label expansion during translation. Capitalize only first word. Use task-based, not system-based labels. Global Navigation
System-Level Error Message [What went wrong]. [What to do now]. Something went wrong. Please try again or contact support. Calm, reassuring. Avoid blame. Prioritize next steps. Max 150 characters Use ARIA role='alert' for real-time announcement Avoid idioms. Use plain, direct instructions. Use when a full process fails (not a field). Always pair with retry guidance or help link. High-Stakes / Blocking Prevents user frustration by clearly showing failure + solution path
Tooltip / Contextual Help Brief explanation of field/functionality This is the legal name used for tax reporting. Helpful, neutral. Avoid full sentences. Max 80 characters Must be dismissible. Include aria-describedby reference. Avoid technical terms that don't translate easily. Use only for non-critical clarification. Don’t hide important decisions behind tooltips. Supplementary Supports self-service learning without overwhelming the screen
Loading State / Spinner Microcopy [Action verb] + [object or state] Loading payments... Calm, transparent. Acknowledge progress. Max 40 characters Use ARIA role='status' and announce visually hidden progress Ensure progress verbs translate well Use when action takes more than 800ms. Show spinner + verb. Avoid humor. Passive/Progress Reduces uncertainty during wait times and builds trust in system behavior
